# Docs-AI — API & MCP Contract

## Overview

The application exposes two interfaces:

1. **REST API** at `/api/*` — consumed by the Next.js frontend
2. **MCP Server** at `/mcp` — consumed by AI assistants via Streamable HTTP transport

Both interfaces read from the same SQLite database and serve files from the same originals directory. There is no authentication layer — Tailscale provides network-level access control.

---

## REST API

All endpoints return JSON unless otherwise noted. Error responses use the standard shape described at the end of this document.

### GET /api/search

Full-text search across all documents.

**Query Parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `q` | string | yes | — | Search query (FTS5 syntax) |
| `page` | number | no | 1 | Page number (1-indexed) |
| `limit` | number | no | 20 | Results per page (max 100) |

**Response 200:**

```json
{
    "results": [
        {
            "id": 42,
            "title": "Deutsche Telekom Invoice — March 2026",
            "description": "Monthly mobile service invoice from Deutsche Telekom for the billing period March 2026.",
            "document_date": "2026-03-15",
            "added_at": "2026-03-20T14:30:00Z",
            "original_filename": "scan_20260315.pdf",
            "mime_type": "application/pdf",
            "snippet": "...monthly <mark>invoice</mark> for mobile service from <mark>Telekom</mark>...",
            "thumbnail_url": "/api/documents/42/thumbnail"
        }
    ],
    "total": 12,
    "page": 1,
    "limit": 20,
    "query": "invoice telekom"
}
```

**Notes:**

- The `snippet` field contains HTML with `<mark>` tags around matching terms, generated by FTS5's `snippet()` function
- Results are ordered by BM25 relevance score
- An empty `q` parameter returns a 400 error; use `GET /api/documents` for listing

### GET /api/documents/:id/original

Serve the original file for download.

**Response 200:**

- `Content-Type`: the document's MIME type
- `Content-Disposition`: `attachment; filename="scan_20260315.pdf"` (original filename)
- Body: the raw file bytes

**Response 404:** standard error shape

---

### GET /api/documents/:id/thumbnail

Serve the thumbnail image for a document.

**Response 200:**

- `Content-Type`: `image/jpeg`
- Body: the thumbnail image bytes
- Cache headers: `Cache-Control: public, max-age=86400` (thumbnails are immutable once generated)

**Response 404:** standard error shape (no thumbnail available)

## Error Response Shape

All error responses use a consistent format:

```json
{
    "error": "error_code",
    "message": "Human-readable description of what went wrong"
}
```

Standard error codes:

| Code | HTTP Status | Description |
|------|-------------|-------------|
| `not_found` | 404 | Resource does not exist |
| `bad_request` | 400 | Invalid parameters |
| `internal_error` | 500 | Unexpected server error |

## MCP Server

The MCP server is mounted at `/mcp` using Streamable HTTP transport. It exposes tools for document access and resources for direct document retrieval.

### Transport

- **URL:** `http://<host>:3000/mcp`
- **Transport:** Streamable HTTP (recommended by MCP spec for remote servers)
- **Authentication:** None (Tailscale network access is the security boundary)

### Tools

MCP tools should be **bulk-first** wherever the operation naturally supports it. For read-oriented tools, the canonical input is an array (`ids`), even if a convenience `id` form is also accepted. Tool outputs should likewise normalize to arrays so AI assistants can fetch multiple documents in one round trip.

#### search_documents

Search the document archive using full-text search.

**Input Schema:**

```json
{
    "type": "object",
    "properties": {
        "query": {
            "type": "string",
            "description": "Search query. Supports natural language queries — the system uses FTS5 with porter stemming for flexible matching."
        },
        "limit": {
            "type": "number",
            "description": "Maximum number of results to return.",
            "default": 10
        }
    },
    "required": ["query"]
}
```

**Output:** JSON array of matching documents:

```json
[
    {
        "id": 42,
        "title": "Deutsche Telekom Invoice — March 2026",
        "description": "Monthly mobile service invoice from Deutsche Telekom.",
        "document_date": "2026-03-15",
        "added_at": "2026-03-20T14:30:00Z",
        "original_filename": "scan_20260315.pdf",
        "snippet": "...monthly invoice for mobile service from Telekom..."
    }
]
```

---

#### get_document

Fetch complete metadata and extracted text for one or more documents.

**Input Schema:**

```json
{
    "type": "object",
    "properties": {
        "ids": {
            "type": "array",
            "items": { "type": "number" },
            "description": "Document IDs to fetch. Preferred over `id` for bulk reads."
        },
        "id": {
            "type": "number",
            "description": "Single document ID convenience form. Equivalent to passing `ids: [id]`."
        }
    }
}
```

**Output:** JSON array of full document records including all metadata fields and the extracted text content. Results are returned in the same order as the requested IDs when possible. This is the primary tool for reading documents found via search.

```json
[
    {
        "id": 42,
        "title": "Deutsche Telekom Invoice — March 2026",
        "description": "Monthly mobile service invoice from Deutsche Telekom for the billing period March 2026.",
        "document_date": "2026-03-15",
        "added_at": "2026-03-20T14:30:00Z",
        "original_filename": "scan_20260315.pdf",
        "stored_filename": "a1b2c3d4e5f6_scan_20260315.pdf",
        "mime_type": "application/pdf",
        "file_size": 245760,
        "page_count": 2,
        "content": "Deutsche Telekom\nInvoice\n..."
    }
]
```

#### get_document_content

Return only the full extracted text of one or more documents.

**Input Schema:**

```json
{
    "type": "object",
    "properties": {
        "ids": {
            "type": "array",
            "items": { "type": "number" },
            "description": "Document IDs to fetch content for. Preferred over `id` for bulk reads."
        },
        "id": {
            "type": "number",
            "description": "Single document ID convenience form. Equivalent to passing `ids: [id]`."
        }
    }
}
```

**Output:** JSON array of content objects. Lighter than `get_document` when only the extracted text is needed — useful for feeding multiple documents into further processing or summarization.

```json
[
    {
        "id": 42,
        "content": "Deutsche Telekom\nInvoice\n..."
    },
    {
        "id": 43,
        "content": "Lease agreement\n..."
    }
]
```

#### list_recent_documents

List recently added documents.

**Input Schema:**

```json
{
    "type": "object",
    "properties": {
        "limit": {
            "type": "number",
            "description": "Maximum number of documents to return.",
            "default": 10
        },
        "since": {
            "type": "string",
```

**Output:** JSON array of document summaries (id, title, description, document_date, added_at), sorted by `added_at` descending.

Get download URLs for one or more original document files.

**Input Schema:**

```json
{
    "type": "object",
    "properties": {
        "ids": {
            "type": "array",
            "items": { "type": "number" },
            "description": "Document IDs to generate download URLs for. Preferred over `id` for bulk reads."
        },
        "id": {
            "type": "number",
            "description": "Single document ID convenience form. Equivalent to passing `ids: [id]`."
        }
    }
}
```

**Output:** JSON array of download objects:

```json
[

        "id": 42,

        "mime_type": "application/pdf",
        "file_size": 245760,
```

The URL is only accessible within the Tailscale network. The AI assistant can present this to the user or fetch it directly.

### Resources

Documents are also exposed as MCP resources for clients that prefer resource-based access.

#### Resource List

`resources/list` returns all documents as resources:

```json
{
    "resources": [
        {
            "uri": "document://42",
            "name": "Deutsche Telekom Invoice — March 2026",
            "description": "Monthly mobile service invoice from Deutsche Telekom.",
            "mimeType": "text/plain"
        }
    ]
}
```

Resource URIs follow the pattern `document://{id}`.

#### Resource Read

`resources/read` for `document://{id}` returns the document metadata and extracted text:

```json
{
    "contents": [
        {
            "uri": "document://42",
            "mimeType": "text/plain",
            "text": "Title: Deutsche Telekom Invoice — March 2026\nDate: 2026-03-15\nOriginal: scan_20260315.pdf\n\n---\n\nDeutsche Telekom\nInvoice\n\nCustomer number: 12345678\n..."
        }
    ]
}
```

The resource text format combines metadata headers with the full extracted content, separated by a horizontal rule. This gives an AI assistant complete context in a single read.

## Implementation Notes

### Next.js API Routes

REST endpoints are implemented as Next.js route handlers under `app/api/`:

```
app/api/
  health/route.ts
  search/route.ts
  documents/route.ts
  documents/[id]/route.ts
  documents/[id]/original/route.ts
  documents/[id]/thumbnail/route.ts
  queue/route.ts
  queue/[id]/retry/route.ts
```

### MCP Server Integration

The MCP server uses the `@modelcontextprotocol/sdk` package with Streamable HTTP transport. It's mounted at `/mcp` as a Next.js route handler that delegates to the MCP SDK's request handling.

The MCP tools share the same database query functions as the REST API — they're thin wrappers that translate between MCP tool schemas and the underlying Kysely queries. For bulk-first tools like `get_document`, `get_document_content`, and `download_document`, the implementation should batch lookups with `WHERE id IN (...)` queries instead of issuing one query per requested ID.
